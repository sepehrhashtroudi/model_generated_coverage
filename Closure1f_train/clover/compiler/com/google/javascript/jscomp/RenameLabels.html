33    /**
34    * RenameLabels renames all the labels so that they have short names, to reduce
35    * code size and also to obfuscate the code.
36    *
37    * Label names have a unique namespace, so variable or function names clashes
38    * are not a concern, but keywords clashes are.
39    *
40    * Additionally, labels names are only within the statements include in the
41    * label and do not cross function boundaries. This means that it is possible to
42    * create one label name that is used for labels at any given depth of label
43    * nesting. Typically, the name "a" will be used for all top-level labels, "b"
44    * for the next nested label, and so on. For example:
45    *
46    * <code>
47    * function bar() {
48    * a: {
49    * b: {
50    * foo();
51    * }
52    * }
53    *
54    * a: {
55    * b: break a;
56    * }
57    * }
58    * </code>
59    *
60    * The general processes is as follows: process() is the entry point for the
61    * CompilerPass, and from there a standard "ScopedCallback" traversal is done,
62    * where "shouldTraverse" is called when descending the tree, and the "visit" is
63    * called in a depth first manner. The name for the label is selected during the
64    * decent in "shouldTraverse", and the references to the label name are renamed
65    * as they are encountered during the "visit". This means that if the label is
66    * unreferenced, it is known when the label node is visited, and, if so, can be
67    * safely removed.
68    *
69    * @author johnlenz@google.com (John Lenz)
70    */
